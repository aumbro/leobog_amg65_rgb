# สรุปการ Reverse Engineer และควบคุมไฟ LEOBOG AMG65

วันที่ทดสอบ: 21 กรกฎาคม 2026  
ระบบที่ใช้ทดสอบ: Windows, เชื่อมต่อคีย์บอร์ดด้วยสาย USB  
โปรแกรมหลัก: `amg65-rgb/amg65_rgb.py`

## 1. ผลลัพธ์ที่ทำได้

โปรแกรมสามารถควบคุมระบบไฟของ AMG65 ได้โดยตรงผ่าน USB HID โดยไม่ต้องแฟลชหรือแก้เฟิร์มแวร์ ได้แก่

- เปลี่ยนเอฟเฟกต์ RGB ของคีย์บอร์ด
- กำหนดสีแบบรายปุ่ม
- สุ่มสีรายปุ่มต่อเนื่อง
- ปิดไฟคีย์บอร์ดทั้งหมด
- ส่งภาพหรือแอนิเมชันไปยัง LED Matrix 63×5
- สตรีม LED Matrix แบบต่อเนื่องด้วย protocol ของโหมด Music
- แสดงนาฬิกา `HH:MM:SS` บนพื้นที่หลัก 56×5
- แสดง Space Invader แบบเคลื่อนไหวบน Matrix ด้านขวา 7×5
- กู้คืนสตรีมอัตโนมัติเมื่อ Windows ปฏิเสธ HID packet ชั่วคราว

สิ่งสำคัญที่สุดที่ค้นพบคือ ลำดับ pixel ในหน่วยความจำของอุปกรณ์ไม่ได้เรียงซ้ายไปขวาตลอด Matrix แต่แบ่งเป็นสี่บล็อก 14×5 แล้วจึงตามด้วยบล็อกขวา 7×5

## 2. โครงสร้าง USB HID

ค่าระบุอุปกรณ์:

| รายการ | ค่า |
|---|---:|
| Vendor ID | `0x0C45` |
| Product ID | `0x800A` |

อินเทอร์เฟซที่ใช้:

| หน้าที่ | Interface | Usage Page | ขนาด report บน Windows |
|---|---:|---:|---:|
| คำสั่งไฟและ Matrix streaming | `MI_02` | `0xFF68` | 65 bytes |
| อัปโหลด Matrix animation ขนาดใหญ่ | `MI_03` | `0xFF67` | 4097 bytes |

ไบต์แรกของ report บน Windows เป็น Report ID และมีค่า `0x00` เสมอ ดังนั้น payload 64 bytes จะถูกวางตั้งแต่ report byte ที่ 1 เป็นต้นไป

คีย์บอร์ดสำหรับการพิมพ์และช่องควบคุมไฟเป็นคนละ HID interface การสตรีม Matrix จึงไม่ได้นำข้อมูลตัวอักษรไปปนกับข้อมูลไฟโดยตรง

## 3. Protocol ไฟคีย์บอร์ดทั่วไป

ลำดับคำสั่งหลักที่ตรวจจาก `DeviceDriver.exe` รุ่น 1.0.3.2:

1. `04 18` — เริ่ม transaction
2. `04 13` — เตรียมข้อมูลเอฟเฟกต์ โดยกำหนด byte 8 เป็น `01`
3. ส่งข้อมูลเอฟเฟกต์ 64 bytes
4. `04 02` — Apply
5. `04 F0` — Finalize

ข้อมูลเอฟเฟกต์ประกอบด้วย:

| Offset ใน payload | ความหมาย |
|---:|---|
| 0 | หมายเลขโหมด |
| 1–3 | Red, Green, Blue |
| 8 | ใช้สีแบบ Colorful หรือไม่ |
| 9 | Brightness ระดับ 0–5 |
| 10 | Speed ระดับ 0–5 |
| 11 | Direction 0–3 |
| 14–15 | Trailer `AA 55` |

โหมดที่รองรับในโปรแกรม:

| ชื่อคำสั่ง | หมายเลขโหมด |
|---|---:|
| `off` | 0 |
| `static` | 1 |
| `single-on` | 2 |
| `single-off` | 3 |
| `glittering` | 4 |
| `falling` | 5 |
| `colourful` | 6 |
| `breath` | 7 |
| `spectrum` | 8 |
| `outward` | 9 |
| `scrolling` | 10 |
| `rolling` | 11 |
| `rotating` | 12 |
| `explode` | 13 |
| `launch` | 14 |
| `ripples` | 15 |
| `flowing` | 16 |
| `pulsating` | 17 |
| `tilt` | 18 |
| `shuttle` | 19 |

## 4. Protocol ไฟรายปุ่ม

ไฟรายปุ่มใช้ transaction แบบ live preview:

1. ส่ง `04 20`
2. กำหนด byte 8 เป็น `08`
3. ส่งตารางสีรวม 512 bytes แบ่งเป็น 8 reports × 64 bytes

แต่ละรายการใช้ 4 bytes:

```text
[light_index, red, green, blue]
```

ตำแหน่งในตารางไม่ใช่ `light_index × 4` แต่ต้องเรียงตามลำดับ `LIGHT_ORDER` ที่ถอดจากโปรแกรมทางการ ปุ่มที่ไม่ระบุสีจะถูกกำหนดเป็น `(0, 0, 0)`

Custom Light เป็นข้อมูลแบบ live preview เฟิร์มแวร์อาจดับไฟเมื่อหยุดส่ง โปรแกรมจึงมี `--hold` และโหมด `random` ซึ่งส่งข้อมูลซ้ำต่อเนื่อง

## 5. โครงสร้าง LED Matrix

จำนวน pixel รวม:

```text
63 columns × 5 rows = 315 RGB pixels
```

พื้นที่ทางกายภาพแบ่งเป็น:

```text
Main panel:   56 × 5 = 280 pixels
Right panel:   7 × 5 =  35 pixels
Total:        63 × 5 = 315 pixels
```

### Mapping ที่ถูกต้อง

ลำดับ raw ของอุปกรณ์คือ:

```text
raw   0– 69  = บล็อกหลักที่ 1 ขนาด 14×5 แบบ row-major
raw  70–139  = บล็อกหลักที่ 2 ขนาด 14×5 แบบ row-major
raw 140–209  = บล็อกหลักที่ 3 ขนาด 14×5 แบบ row-major
raw 210–279  = บล็อกหลักที่ 4 ขนาด 14×5 แบบ row-major
raw 280–314  = บล็อกขวาขนาด 7×5 แบบ row-major
```

สูตรแปลง raw index เป็นพิกัด `(x, y)`:

```python
if raw_index < 280:
    block = raw_index // 70
    inside = raw_index % 70
    x = block * 14 + inside % 14
    y = inside // 14
else:
    inside = raw_index - 280
    x = 56 + inside % 7
    y = inside // 7
```

ตารางย่อ:

| ช่วง raw | ช่วง X | Y | รูปแบบ |
|---:|---:|---:|---|
| 0–69 | 0–13 | 0–4 | 14 pixels ต่อแถว |
| 70–139 | 14–27 | 0–4 | 14 pixels ต่อแถว |
| 140–209 | 28–41 | 0–4 | 14 pixels ต่อแถว |
| 210–279 | 42–55 | 0–4 | 14 pixels ต่อแถว |
| 280–314 | 56–62 | 0–4 | 7 pixels ต่อแถว |

นี่คือเหตุผลที่การส่งไฟไล่ raw index ดูเหมือนจุดกระโดด: เมื่อครบ 14 จุด จุดถัดไปจะกลับไปต้นบล็อกแล้วเลื่อนลงหนึ่งแถว ไม่ได้วิ่งต่อไปตลอดความกว้าง 63 จุด

## 6. วิธีหา Mapping จากวิดีโอ

ใช้วิดีโอ `WIN_20260721_20_56_47_Pro.mp4` ซึ่งมีข้อมูลประมาณ:

- ความละเอียด 1280×720
- 29.97 FPS
- 1,991 frames
- ความยาวประมาณ 66.43 วินาที

โปรแกรม calibration ส่งไฟทีละ raw pixel ที่ประมาณ 5 FPS และวนครบ 315 จุด โดยใช้สี marker:

```text
raw_index หาร 35 ลงตัว = เขียว
raw_index หาร 7 ลงตัว  = น้ำเงิน
ตำแหน่งอื่น              = แดง
```

ขั้นตอนวิเคราะห์:

1. Crop เฉพาะบริเวณ Matrix จากแต่ละ video frame
2. แปลงภาพเป็น HSV
3. Threshold pixel ที่มี saturation และ brightness สูง
4. ใช้ connected components หา centroid ของจุดที่ติด
5. รวม frame ต่อเนื่องที่เป็นจุดเดียวกันเป็นหนึ่ง run
6. ใช้ marker สีน้ำเงินทุก 7 จุดและสีเขียวทุก 35 จุดตรวจลำดับเวลา
7. เปรียบเทียบ centroid ของแต่ละ raw index กับแถวและคอลัมน์จริง

ตรวจจับได้ 329 runs จากวิดีโอซึ่งครอบคลุมรอบเต็ม 315 จุดและจุดต้นรอบถัดไป ผลที่เห็นชัดคือกลุ่ม raw 70 จุดจะสร้างพื้นที่ 14×5 หนึ่งบล็อก จนครบสี่บล็อก แล้ว 35 จุดสุดท้ายสร้างพื้นที่ 7×5 ด้านขวา

ก่อนวิเคราะห์วิดีโอเคยทดลอง mapping แบบ 8×5 หลายบล็อก ทำให้นาฬิกาและตัวอักษรบิดเบี้ยว Mapping ดังกล่าวถูกยกเลิกแล้ว

## 7. Protocol อัปโหลด Matrix Animation

การอัปโหลดภาพหรือแอนิเมชันแบบเก็บเป็นชุดใช้ทั้งสองอินเทอร์เฟซ:

1. ส่ง `04 18` ทาง `MI_02`
2. ส่ง `04 33` ทาง `MI_02`
3. ใส่จำนวน chunk ที่ byte 8
4. ส่งข้อมูลผ่าน report ขนาด 4097 bytes ทาง `MI_03`
5. หน่วงแต่ละ large report ประมาณ 170 ms
6. ส่ง `04 02`
7. ส่ง `04 F0`

โครงสร้าง animation payload:

```text
[frame_count, 00, 0C, 00]
[RGB ของทุก pixel ของทุก frame]
[AA, 55]
```

กรณีภาพเดียว โปรแกรมทางการมี padding `00` ก่อน `AA 55`

วิธีนี้เหมาะกับภาพหรือ animation ที่อัปโหลดเป็นชุด แต่ไม่เหมาะกับภาพสดที่ต้องเปลี่ยนตลอดเวลา เพราะ report ใหญ่และมี delay สูง

### 7.1 ผลทดสอบเส้นทางอัปโหลด (22 กรกฎาคม 2026)

หลังพบว่า live stream ตันที่ 5-6 FPS (§9.3) เส้นทางนี้กลายเป็นทางเดียวที่ได้ภาพลื่น
จึงทดสอบขีดจำกัดจริงด้วยการยิง animation ที่แต่ละ frame เป็นตัวเลขต่างกัน (อ่านผลด้วยตาได้ทันที)

| frame | ขนาด payload | chunk | ผล |
|---:|---:|---:|---|
| 1 | 952 | 1 | ผ่าน |
| 4 | 3,786 | 1 | ผ่าน |
| 8 | 7,566 | 2 | ผ่าน |
| 32 | 30,246 | 8 | ผ่าน |
| 64 | 60,486 | 15 | ผ่าน |
| 69 | 65,211 | 17 | ผ่าน |
| 70 | 66,156 | 17 | ผ่าน |
| 76 | 71,826 | 18 | ผ่าน |
| 83 | 78,441 | 20 | ผ่าน |
| **255** | **240,981** | **59** | **ผ่าน** |

**ไม่มีเพดานจำนวน frame ฝั่งฮาร์ดแวร์** — 255 คือค่าสูงสุดที่ส่งได้อยู่แล้ว
เพราะจำนวน frame เก็บใน byte เดียวของ header เกินกว่านี้ส่งไม่ได้ตั้งแต่ต้นทาง
เพดานสเปกกับเพดานฮาร์ดแวร์จึงเป็นตัวเดียวกัน ค่า `MAX_FRAMES = 255` ในโค้ดยืนยันแล้ว

ความจุที่ได้จริง: 255 frame ที่ราว 10 FPS = animation ยาว **~25 วินาที** วนอยู่ในคีย์บอร์ด
โดยไม่ต้องต่อคอมเลย

สมมติฐานที่ตกไปทั้งหมด:

- stride ระหว่าง frame ผิด — ไม่ใช่ วางชิดกัน 945 ไบต์ต่อ frame ถูกแล้ว
- ส่งหลาย chunk ไม่ติด — ไม่ใช่ 20 chunk ยังต่อกันได้ปกติ
- มีเพดานจำนวน frame (เดา 64KB = 69 frame) — ไม่ใช่ 70 ขึ้นไปก็ยังผ่าน
- เฟิร์มแวร์เล่นเร็วเกินจนภาพเลื่อนเบลอ — ไม่ใช่ ข้อความวิ่งกว้าง 98 px 83 frame อ่านออกปกติ
- ข้อมูลที่เบคจาก GIF เพี้ยน — ไม่ใช่ ตรวจแล้วมี 2 สีล้วน เลื่อน 1 px ต่อ frame เป๊ะทั้ง 83 frame

⚠️ **ความผิดปกติที่พบครั้งเดียวแล้วไม่ซ้ำ**: การอัปโหลด animation หลาย frame *ครั้งแรกสุด*
ต่อจากการอัปโหลดภาพนิ่ง 1 frame ทันที ได้ภาพที่ตัวอักษรมั่ว
อัปโหลดไฟล์เดิมซ้ำอีกครั้งได้ภาพถูกต้อง และไม่เกิดอีกเลยตลอดการทดสอบที่เหลือ
เดาว่าเฟิร์มแวร์ไม่ได้เคลียร์บัฟเฟอร์เก่าเมื่อจำนวน frame เปลี่ยนจาก 1 เป็นหลาย frame
**ถ้าเจออาการนี้ให้อัปโหลดซ้ำอีกครั้ง**

ข้อดีที่ยืนยันแล้ว: ตลอดการทดสอบทั้งวัน **`MI_03` ไม่เคยค้างเลยสักครั้ง** แม้ในช่วงที่
`MI_02` ค้างสนิท (ค้างไป 4 รอบ) เพราะการอัปโหลดเป็นการส่งครั้งเดียวจบ ไม่ใช่การยิงต่อเนื่อง

## 8. Protocol Matrix Streaming แบบ Music

การสตรีมสดใช้เฉพาะ report ขนาด 65 bytes ทาง `MI_02` ลำดับต่อหนึ่ง frame คือ:

1. `04 18`
2. รอ 8.5 ms
3. `04 35` โดย byte 8 = `0F`
4. รอ 8.5 ms
5. ส่ง RGB payload 15 reports × 64 bytes
6. รอ 8.5 ms ระหว่าง reports
7. ส่ง zero flush report 64 bytes อีกหนึ่ง report
8. รอ 8.5 ms
9. ส่ง `04 02`

ข้อมูลภาพต่อ frame:

```text
315 pixels × 3 bytes = 945 bytes
Trailer AA 55        =   2 bytes
Padding              =  13 bytes
รวม                   = 960 bytes = 15 × 64
```

ค่าช่องสีถูกจำกัดสูงสุดไว้ที่ `0xFE` หรือ 254 เพราะ protocol ของโปรแกรมทางการไม่ใช้ `0xFF` ใน live stream

จำนวน HID reports ต่อ frame:

```text
Begin 1 + Init 1 + RGB 15 + Flush 1 + Apply 1 = 19 reports
```

## 9. FPS และแบนด์วิดท์

ค่าที่ใช้ตอนแรก:

| ค่า | ผลที่เห็น |
|---:|---|
| 10 FPS / delay สั้นประมาณ 4 ms | มีอาการกระพริบ |
| เป้าหมาย 8 FPS / delay ประมาณ 7 ms | ทำงานได้ช่วงหนึ่ง แต่พบกระพริบภายหลัง |
| เป้าหมาย 7 FPS / delay 8.5 ms | เสถียรกว่าและใช้เป็นค่าปัจจุบัน |

### 9.1 ผลวัดจริงจาก bench_fps.py (22 กรกฎาคม 2026)

วัดสองแกนพร้อมกัน: จำนวน report ต่อ frame และ delay ระหว่าง packet
สูตรที่ตัด report ออกคือตัด `04 18` / `04 35` ที่ส่งซ้ำทุก frame (ส่งครั้งเดียวตอนเข้าโหมดแทน)
และตัด zero flush report

| สูตร | reports/frame | delay 8.5 ms | 2.0 ms | 1.0 ms | 0 ms |
|---|---:|---:|---:|---:|---:|
| ของเดิม | 19 | 5.3 | 13.6 | 18.6 | 49.7 |
| ตัด flush | 18 | 5.6 | 15.2 | 21.0 | 53.7 |
| ตัด header | 17 | 6.0 | 15.3 | 20.5 | 52.9 |
| ตัดทั้งคู่ | 16 | 6.3 | 16.5 | 21.9 | 55.7 |

หน่วยเป็น FPS **write ไม่พลาดเลยแม้แต่ครั้งเดียวในทุกช่อง**

ข้อสรุป:

- คอขวดคือ `time.sleep()` ระหว่าง packet ไม่ใช่จำนวน packet
  ตัด report 19 → 16 ได้เพิ่มราว 12% แต่ตัด delay ได้เพิ่มสิบเท่า
- `04 18` และ `04 35` ไม่ใช่ handshake ต่อ frame ส่งครั้งเดียวตอนเข้าโหมดก็พอ
  แต่กำไรน้อยจนไม่ใช่ประเด็นหลัก
- `time.sleep(0.001)` บน Windows กินจริงราว 2.8 ms เพราะ timer resolution หยาบ
  ค่า delay ต่ำจึงไม่ได้ผลเป็นเส้นตรงตามที่ตั้ง

### 9.2 delay ต่ำทำให้ endpoint ค้าง ไม่ใช่แค่ภาพกระพริบ

รันการทดลองเดิมซ้ำรอบที่สอง ได้ผลต่างจากรอบแรกอย่างมีนัยสำคัญ:

| รอบ | 8.5 ms | 6 ms | 4 ms | 2 ms | 1 ms | 0 ms |
|---|---:|---:|---:|---:|---:|---:|
| ที่ 1 | 6.3 | 8.2 | 11.1 | 18.4 | 25.5 | 59.9 |
| ที่ 2 | 6.3 | 8.3 | 10.9 | 16.6 | **14.1 + frame หลุด → ค้าง** | ไปไม่ถึง |

รอบที่สองที่ delay 1.0 ms มี frame หลุดหนึ่ง frame (FPS ตกจาก 25.5 เหลือ 14.1 เพราะเสียเวลา retry)
แล้ว `MI_02` ค้างถาวรทันที ส่งอะไรต่อไม่ได้อีกเลยจนกว่าจะถอดสาย
`doctor` หลังจากนั้นยืนยัน: `MI_02` write -1 ที่ 1015 ms ขณะที่ `MI_03` ยังปกติ

ข้อสรุปที่ใช้งานได้จริง:

- **การค้างเป็นแบบสุ่ม ไม่ใช่เส้นตายที่ค่าใดค่าหนึ่ง** รอบแรกลง 0 ms ได้โดยไม่มีอะไรเสียหาย
  รอบสองพังตั้งแต่ 1 ms การกวาดสั้น ๆ รอบเดียวจึงตัดสินไม่ได้
- นี่คือสาเหตุที่แท้จริงของอาการ "สตรีมหยุดกลางทาง" ตั้งแต่เฟสแรก ไม่ใช่ HID endpoint busy ชั่วคราว
- กลไก: ส่งถี่กว่าที่อุปกรณ์ระบายได้ คิว output ล้น จน overlapped transfer ค้างคาไม่มีวันจบ
- **retry และ reopen handle ไม่ช่วยเลย** เพราะการเปิด handle ใหม่ไม่ล้างคิวของ endpoint
  ทำได้แค่เสียเวลา 1 วินาทีต่อครั้งที่พยายาม โค้ดจึงต้องแยกอาการนี้ออกจาก write ที่พลาดชั่วคราว
  แล้วเลิกทันที (`EndpointStalled` ใน `amg65/device.py` ใช้เกณฑ์ write ที่บล็อกครบ 0.9 วินาที)

วิธีหาค่า delay ที่ใช้ได้จริง: ทดสอบทีละค่ายาว ๆ ไม่ใช่กวาดเร็ว ๆ

```bash
python bench_fps.py --soak 4 --seconds 180
```

### 9.3 แยก delay ของข้อมูลออกจาก delay ของคำสั่ง — และข้อสรุปสุดท้าย

19 packet ต่อ frame ทำงานคนละอย่าง จึงลองแยกค่าหน่วง:

| ชนิด | packet | หน้าที่ |
|---|---|---|
| คำสั่ง | `04 18`, `04 35`, flush, `04 02` | เปลี่ยนสถานะและ apply ทั้ง frame |
| ข้อมูล | RGB 15 reports | ยัดข้อมูลลงบัฟเฟอร์ |

ตรึงคำสั่งไว้ที่ 8.5 ms แล้วไล่ลดเฉพาะข้อมูล:

| delay ข้อมูล | FPS | ค้าง | ตาเห็น dot ผิดตำแหน่ง |
|---:|---:|---|---|
| 8.5 ms | 5.3 | ไม่ | **สะอาด** |
| 6.0 | 6.5 | ไม่ | มี |
| 4.0 | 8.3 | ไม่ (ใน 1 นาทีแรก) | มี |
| 2.0 | 10.8 | ค้างที่วินาทีที่ 12 | มี |
| 0.0 | 18.3 | ค้างทุกครั้ง (3/3) | มี |

**ข้อสรุป: 8.5 ms คือค่าเดียวที่ภาพสะอาดจริง เพดาน live stream ของฮาร์ดแวร์นี้คือราว 5–6 FPS**
การแยก delay ได้ FPS เพิ่มจริงในเชิงตัวเลข แต่ทุกค่าที่เร็วกว่า 8.5 ms มี dot ผิดตำแหน่ง
จึงใช้งานจริงไม่ได้ ปิดเรื่องการเร่ง FPS ทางนี้

สังเกตสำคัญ: **dot ผิดตำแหน่งกับ endpoint ค้างคืออาการเดียวกันคนละความรุนแรง**
ทั้งคู่เกิดจากเครื่องรับไม่ทันแล้วทำ report หาย
หายก้อนเดียว → ข้อมูลเลื่อน 64 ไบต์ = 21.3 pixel แล้ว frame ถัดไปกลับมาปกติ
หายตอนคิวล้นจริง → ค้างถาวร
ดังนั้น **dot ผิดตำแหน่งคือสัญญาณเตือนล่วงหน้าว่ากำลังจะค้าง** ค่าที่ปลอดภัยต้องไม่มี dot หลงเลย

บทเรียนเรื่องวิธีทดสอบ: **การกวาดสั้น ๆ ให้ผลลวงไปแล้วสองครั้ง**
รอบแรก `--visual` ผ่านลงถึง 0 ms ทั้งที่รอบสองพังที่ 1 ms
และ `--split` ผ่านช่วง 2 ms มาได้เพราะช่วงละ 8 วินาที ขณะที่ soak จริงพบว่าค้างที่วินาทีที่ 12

⚠️ ข้อควรระวังอีกข้อ: **throughput ผ่านไม่ได้แปลว่าภาพไม่กระพริบ**
การกระพริบเกิดฝั่งเฟิร์มแวร์ที่เอา frame ซึ่งยังมาไม่ครบไปแสดง ซึ่งไม่ทำให้ `hid_write` ล้มเหลว
โปรแกรมจึงตรวจไม่ได้ ต้องยืนยันด้วยตาผ่าน `python bench_fps.py --visual`

แบนด์วิดท์โดยประมาณที่ 7 FPS:

```text
19 reports × 65 bytes × 7 ≈ 8,645 bytes/second
```

ปริมาณนี้ต่ำมากเมื่อเทียบกับแบนด์วิดท์ USB โดยรวม และข้อมูลการพิมพ์อยู่คนละ HID interface จึงไม่ควรกินแบนด์วิดท์จนทำให้คีย์หาย อย่างไรก็ตาม Windows HID endpoint อาจ busy ชั่วคราวและปฏิเสธ output report ได้ ซึ่งเป็นสาเหตุที่สตรีมรอบแรกหยุด

## 10. การแก้อาการสตรีมหลุด

อาการที่พบ:

```text
OSError: ส่งคำสั่ง Matrix ไม่ครบ 65 ไบต์
```

สาเหตุโดยตรงคือ `hid.write()` คืนค่าจำนวน bytes ไม่ครบ 65 หรือเกิด `OSError` โปรแกรมเดิมโยน exception แล้วปิด process ทำให้ภาพถูกต้องช่วงแรกก่อนจะหายไป

การแก้ปัจจุบันมีสองระดับ:

1. Retry report เดิมสูงสุด 3 ครั้ง โดยหน่วง 20 ms ระหว่างครั้ง
2. ถ้ายังไม่สำเร็จ ให้ปิดและเปิด `MI_02` ใหม่ แล้วส่ง frame ทั้งชุดใหม่ตั้งแต่ `04 18`

ถ้าถอดสาย โปรแกรมจะวนรอและพยายามเปิด endpoint ใหม่ทุก 250 ms เมื่อเสียบกลับจึงสามารถกลับมาส่งต่อได้ โดย Matrix live stream ใช้ `MI_02` เป็นหลัก

### 10.1 อาการ MI_02 ค้างทั้งช่อง (22 กรกฎาคม 2026)

พบสภาพที่ `hid_write` ไปยัง `MI_02` **ค้างครบ timeout 1 วินาทีแล้วคืน -1 ทุกครั้ง** โดยมี error

```text
hid_write/WaitForSingleObject: (0x000003E5) Overlapped I/O operation is in progress
```

สิ่งที่ตรวจแล้วว่า *ไม่ใช่* สาเหตุ:

- อุปกรณ์ยังอยู่ครบ `Get-PnpDevice` เห็นทุก interface สถานะ OK และเป็น `USB\VID_0C45&PID_800A`
  ไม่ใช่ `BTHENUM` แปลว่าต่อผ่านสาย
- `hid.enumerate` เจอ `MI_02` ปกติ และ `open_path` สำเร็จ อ่าน manufacturer/product ได้
- ไม่มีโปรแกรม LEOBOG ทางการหรือ process Python ค้างถือ handle อยู่

จุดที่แยกโรคได้: **`MI_03` เขียนผ่านปกติ (4097 ไบต์ / 60 ms) ในขณะที่ `MI_02` ค้าง**
แปลว่าอุปกรณ์ยังตอบสนอง เป็นอาการ interrupt OUT endpoint ของ interface 2 ค้างอย่างเดียว
สาเหตุที่เข้าเค้าที่สุดคือ stream ถูกฆ่ากลาง frame ทิ้ง transfer ค้างไว้

วิธีแก้ที่ได้ผล: **ถอดสาย USB เสียบใหม่** (การ disable/enable ผ่าน PnP ก็ควรได้ผลเหมือนกันแต่ต้องสิทธิ์ admin)

ตรวจสภาพช่องทั้งสองได้ด้วย

```bash
python -m amg65 doctor
```

## 14. ข้อควรระวังและข้อจำกัด

- ต้องเชื่อมต่อด้วยสาย USB; Bluetooth หรือ 2.4 GHz อาจไม่เปิด vendor HID interface เดียวกัน
- ควรปิดโปรแกรม LEOBOG ทางการ เพราะทั้งสองโปรแกรมอาจส่งข้อมูลไป endpoint เดียวกันและแย่งโหมดไฟกัน
- หากเปลี่ยน USB port หรือ power cycle ควรตรวจ VID/PID และ interface อีกครั้ง
- FPS สูงเกินไปทำให้กระพริบหรือเกิด HID write failure
- `--fps` เป็นเป้าหมายระดับ animation แต่ความเร็วจริงถูกจำกัดด้วย packet delay ของ protocol
- การสตรีม Matrix ทำให้ต้องมี process ทำงานค้างอยู่ เมื่อหยุด process ภาพ live อาจค้างชั่วคราวหรือกลับไปเอฟเฟกต์เดิมของเฟิร์มแวร์
- Mapping นี้ยืนยันกับบอร์ด AMG65 เครื่องที่ทดสอบ หากมี hardware revision อื่นควรรัน dot calibration ตรวจอีกครั้ง

## 16. สถานะล่าสุด

- Mapping 63×5: ถูกต้อง
- นาฬิกา 56×5 และ Space Invader 7×5: แสดงถูกต้อง
- Matrix streaming: ทำงานได้ ไม่มี frame หลุดในการวัดรอบล่าสุด
- Retry/reconnect อัตโนมัติ: เพิ่มแล้ว
- เรื่อง FPS: **ปิดจ๊อบแล้ว** — 8.5 ms คือค่าเดียวที่ภาพสะอาด เพดานจริงราว 5-6 FPS
  ค่าที่เร็วกว่านั้นได้ FPS เพิ่มจริงแต่มี dot ผิดตำแหน่งทุกค่า (§9.3)
- ค่าเริ่มต้นในโปรแกรม: delay 8.5 ms ทั้ง data และ command — ไม่ต้องแก้อะไร
- ทางสำหรับภาพลื่น: อัปโหลดเข้าเครื่องผ่าน `MI_03` — **ทดสอบแล้วใช้งานได้จริง** (§7.1)
  ยิงผ่านถึง 83 frame / 78 KB / 20 chunk ไม่พบเพดานจำนวน frame
  เฟิร์มแวร์เล่นวนเอง ไม่ต้องมี process ค้าง ไม่มีทราฟฟิก USB จึงไม่มีโอกาสค้างเลย
  ใช้กับภาพลูปได้ แต่ใช้กับ scene ที่ต้องการข้อมูลสดไม่ได้
- คำสั่ง `upload` เบคได้ทั้งจากไฟล์ภาพ/GIF และจาก scene ที่มีอยู่ โดยไม่ต้องแก้ scene
  เพราะ `Scene.render()` ไม่ยุ่งกับ HID อยู่แล้ว จะถูกสตรีมหรือถูกเบคก็ไม่รู้เรื่อง
- ยังไม่รู้ความหมายของไบต์ที่ 2 ใน header (`0x0C`) — `--speed` เปิดให้ทดลองค่าอื่นได้
